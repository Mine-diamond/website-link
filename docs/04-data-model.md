# Data Model

## Current Bookmark Model

Current bookmark records use this shape:

```ts
type WebsiteBookmark = {
  id: string;
  title: string;
  url: string;
  tags: string[];
  notes: string;
  favicon: string | null;
  importance: number;
  dateAdded: string;
};
```

## Field Semantics

`id`

Unique bookmark ID. Currently generated by the backend with `Date.now().toString()`.

`title`

User-visible bookmark title.

`url`

Target website URL.

`tags`

Array of user-defined tags. The web UI and extension collect tags from comma-separated text.

`notes`

Free-form note text.

`favicon`

Favicon URL. The backend currently generates a Google favicon service URL from the target hostname when no favicon is provided.

`importance`

Numeric importance level. The UI treats this as a 1 to 5 star value.

`dateAdded`

ISO timestamp created when the bookmark is added.

## Current Storage Shape

All bookmarks are stored as one JSON array under the Cloudflare KV key:

```text
bookmarks
```

Example:

```json
[
  {
    "id": "1720000000000",
    "title": "Example",
    "url": "https://example.com",
    "tags": ["tool"],
    "notes": "Example note",
    "favicon": "https://www.google.com/s2/favicons?domain=example.com&sz=32",
    "importance": 3,
    "dateAdded": "2026-01-01T00:00:00.000Z"
  }
]
```

## Launch Desk Compatibility

Launch Desk has a matching `WebsiteBookmark` type and depends on the current fields.

The most important fields for Launch Desk are:

- `id`, used as `remoteBookmarkId` when pinned locally.
- `title`, used as the local URL item title.
- `url`, used as the local URL item target.
- `favicon`, used as the local URL item icon.
- `tags` and `notes`, used for search matching.
- `importance`, used for search scoring and URL page sorting.
- `dateAdded`, used for sorting and local transient item metadata.

Breaking changes to these fields should be coordinated with Launch Desk.

## Recommended Near-Term Enhancements

Recommended additions:

```ts
type WebsiteBookmark = {
  id: string;
  title: string;
  url: string;
  tags: string[];
  notes: string;
  favicon: string | null;
  importance: number;
  dateAdded: string;
  updatedAt: string;
};
```

Recommended changes:

- Generate IDs with `crypto.randomUUID()`.
- Add `updatedAt` on add and update.
- Validate URL, tags, notes, and importance on the backend.
- Avoid allowing arbitrary fields to overwrite existing bookmark records.

## Future Sync Model

If website-link needs incremental sync, use a sync-friendly shape:

```ts
type SyncBookmark = {
  id: string;
  title: string;
  url: string;
  tags: string[];
  notes: string;
  favicon: string | null;
  importance: number;
  dateAdded: string;
  updatedAt: string;
  deletedAt?: string | null;
  version: number;
};
```

Recommended sync endpoint:

```text
GET /api/bookmarks/sync?since=<timestamp>
```

Recommended update payload:

```json
{
  "id": "bookmark-id",
  "title": "New Title",
  "url": "https://example.com",
  "tags": ["dev"],
  "notes": "note",
  "importance": 4,
  "baseVersion": 7
}
```

Conflict behavior can start with last-write-wins, but version-aware conflict responses should be considered before multi-device editing becomes important.

## Storage Migration Options

The current KV single-array model is simple and suitable for a personal MVP.

Possible future storage options:

- Keep KV but split bookmarks by ID.
- Move structured data to Cloudflare D1.
- Store tags as separate records only if tag querying becomes important.
- Add embeddings only if AI semantic search becomes a concrete feature.

D1 migration should not be required before basic API hardening and style redesign.
